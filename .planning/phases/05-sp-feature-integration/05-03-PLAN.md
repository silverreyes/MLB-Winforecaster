<objective>
Convert season-aggregate SP stats to season-to-date rolling (SP-03), add K-BB% differential (SP-04), WHIP differential (SP-05), ERA differential (SP-06), and implement cold-start handling (SP-10).

Purpose: The v1 FeatureBuilder uses full-season FanGraphs aggregates for SP features, causing temporal leakage (a June game sees September stats). This plan replaces the season-aggregate lookup with per-game rolling computation using cumsum + shift(1), matching the established v1 rolling pattern from `_add_rolling_features()`. It also adds three new high-signal SP differential features and handles the cold-start case for first-start-of-season and rookie pitchers.

Output: Refactored `_add_sp_features()` producing temporally-safe rolling SP differentials, new `sp_k_bb_pct_diff`, `sp_whip_diff`, `sp_era_diff` columns, cold-start imputation, and extended test coverage.
</objective>

```python
# Available: 'Name', 'IDfg', 'ERA', 'FIP', 'xFIP', 'SIERA', 'WHIP', 'K%', 'BB%', 'K-BB%', 'IP', 'GS'
```

Season-to-date rolling formulas:
```
std_era = (cumsum(ER) * 9) / cumsum(IP)   # with shift(1)
std_whip = (cumsum(H + BB)) / cumsum(IP)   # H from hits column or derived
std_k_pct = cumsum(K) / cumsum(BF)         # BF = batters faced
std_bb_pct = cumsum(BB) / cumsum(BF)
std_k_bb_pct = std_k_pct - std_bb_pct      # or use K-BB% directly from game logs
```

NOTE: For season-to-date K-BB%, WHIP, and ERA, the research recommends using FanGraphs season-level data as the COLD-START fallback only. The rolling values come from MLB Stats API per-game logs via cumsum + shift(1).

However, the MLB Stats API gameLog does NOT return hits, batters faced, or WHIP per game directly. The v2 game log has: date, innings_pitched, earned_runs, strikeouts, base_on_balls, home_runs, number_of_pitches, games_started.

**Practical approach for features computable from game log v2:**
- `std_era` = cumsum(ER)*9 / cumsum(IP) with shift(1) -- YES (ER, IP in log)
- `std_k_bb_pct` = (cumsum(K) - cumsum(BB)) / cumsum(BF) -- APPROXIMATE: BF not in log. Use (cumsum(K) - cumsum(BB)) / (cumsum(IP) * 3 + cumsum(K) + cumsum(BB)) as rough BF approximation, OR use the simpler K-BB per IP differential: (K - BB) / IP which is proportional and cancels in differentials.
- `std_whip` = WHIP requires hits. NOT computable from game log v2 (no hits column). Use FanGraphs season-aggregate WHIP for now (same leakage concern) OR use a proxy. **Decision: Use FanGraphs season-level WHIP as a "slow-changing" feature. Since WHIP is relatively stable within a season (not as date-sensitive as ERA), the leakage is minimal. Document this tradeoff.**

**Revised approach:**
- `sp_era_diff`: Season-to-date rolling from game logs (cumsum + shift(1)). Fully temporal-safe.
- `sp_k_bb_pct_diff`: Season-to-date rolling from game logs. Compute as (cumsum(K) - cumsum(BB)) / cumsum(IP) as a K-BB rate per inning, then convert to percentage-like scale. Since we only use differentials, the exact scale cancels out. Alternatively, compute per-game K-BB count, cumsum with shift(1), divide by cumsum(IP).
- `sp_whip_diff`: Use FanGraphs season-level WHIP. Mark this feature as "season-aggregate" in feature_sets.py comments. The WHIP temporal safety test should expect a CONSTANT value per pitcher per season (not game-varying) -- this is acceptable for WHIP since the test discriminates between "accidentally constant" (bug) and "intentionally constant" (design decision).
- `sp_siera_diff`: Keep using FanGraphs season-level (same as v1). SIERA is not computable from game logs.

       logs["prev_cum_er"] = grp["cum_er"].shift(1)

       logs["prev_cum_bb"] = grp["cum_bb"].shift(1)

       # Season-to-date ERA
       logs["std_era"] = np.where(
           logs["prev_cum_ip"] > 0,
           (logs["prev_cum_er"] * 9) / logs["prev_cum_ip"],
           np.nan,
       )

       # Season-to-date K-BB rate (per 9 IP for scale)
       logs["std_k_bb_rate"] = np.where(
           logs["prev_cum_ip"] > 0,

           np.nan,
