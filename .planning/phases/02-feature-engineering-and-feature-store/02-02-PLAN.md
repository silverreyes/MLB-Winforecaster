**Part A: Create `src/features/sp_recent_form.py`** -- Date-range pitching stats loader for SP recent form.

This module provides the 30-day rolling ERA window per CONTEXT.md locked decision: "SP recent form: rolling 30-day ERA window via `pybaseball.pitching_stats_range(start_dt, end_dt)`."

```python
"""SP recent form loader using pybaseball.pitching_stats_range() with caching.

Provides 30-day rolling ERA window for starting pitchers. Each call fetches
FanGraphs pitcher stats for a specific date range and caches to Parquet.

Per CONTEXT.md locked decision: SP recent form uses pitching_stats_range()
for a 30-day ERA window, not season-level ERA.
"""
import time
import logging
import pandas as pd
from datetime import datetime, timedelta
from pybaseball import pitching_stats_range as pybaseball_pitching_stats_range
from src.data.cache import is_cached, save_to_cache, read_cached

logger = logging.getLogger(__name__)
RATE_LIMIT_DELAY = 2.0  # seconds between FanGraphs requests
```

Functions:

1. `fetch_sp_recent_form(game_date: str, season: int) -> pd.DataFrame`:
   - Computes a 30-day window ending the day BEFORE game_date (temporal safety: does not include game day itself)
   - `end_dt = (datetime.strptime(game_date, "%Y-%m-%d") - timedelta(days=1)).strftime("%Y-%m-%d")`
   - `start_dt = (datetime.strptime(end_dt, "%Y-%m-%d") - timedelta(days=30)).strftime("%Y-%m-%d")`
   - Clamp `start_dt` to season opening day if the 30-day window extends before the season start (approximate: use f"{season}-03-20" as earliest reasonable date; if start_dt < that, set start_dt = f"{season}-03-20")
   - Cache key: `sp_form_{game_date}` (one cache entry per game date, shared across all games on that date)
   - Parquet path: `pybaseball/sp_form_{game_date}.parquet`
   - If cached, return `read_cached(key)` immediately
   - Otherwise call `pybaseball_pitching_stats_range(start_dt, end_dt)`
   - Add `game_date` column to the result for join purposes
   - Cache and return
   - On exception, log error and set `result = pd.DataFrame()` (graceful degradation)
   - ALWAYS call `save_to_cache(result, key, parquet_path, season)` before returning, even when `result` is an empty DataFrame. If empty returns are not cached, every re-run re-hits the API for off-days and exception dates, burning rate limit budget unnecessarily.
   - Return `result`

2. `fetch_sp_recent_form_bulk(game_dates: list[str], season: int) -> dict[str, pd.DataFrame]`:
   - Batch fetcher for all unique game dates in a season
   - Deduplicates dates first (many games share the same date)
   - For each unique date: check cache first, only call pybaseball if uncached
   - Rate limit: `time.sleep(RATE_LIMIT_DELAY)` after each uncached fetch
   - Returns dict mapping game_date -> DataFrame of pitcher stats for that 30-day window
   - Log progress: `logger.info(f"Fetched SP form for {date} ({i}/{total_uncached})")`

**IMPORTANT rate limiting note:** The number of unique game dates across 10 seasons is approximately 10 x 180 = 1,800 unique dates. At 2 seconds per call, this is ~60 minutes for a full uncached fetch. This is acceptable for a one-time data ingestion (similar to the 300-call team_game_logs scrape), and all results are cached. Subsequent FeatureBuilder.build() calls read from cache instantly.

**Part B: Create `src/features/feature_builder.py`** with a `FeatureBuilder` class. The class reads raw cached Parquet data from Phase 1 loaders and computes all differential features.

**Class signature:**
```python
class FeatureBuilder:
    def __init__(self, seasons: list[int], as_of_date: str | None = None):
        self.seasons = seasons
        self.as_of_date = as_of_date  # YYYY-MM-DD cutoff for walk-forward

    def build(self) -> pd.DataFrame:
        """Build complete feature matrix. Returns one row per game."""
```

**The `build()` method pipeline (each step is a private method):**

1. `_load_schedule()` -> pd.DataFrame:
   - For each season in self.seasons, call `fetch_schedule(season)`
   - Concatenate all seasons
   - If `self.as_of_date` is set, filter `game_date < self.as_of_date`
   - Ensure `game_date` is datetime type

2. `_filter_tbd_starters(df)` -> pd.DataFrame:
   - Drop rows where `home_probable_pitcher` is None/NaN OR `away_probable_pitcher` is None/NaN
   - Log count of excluded rows: `logger.info(f"Excluded {n} games with TBD starters")`

3. `_add_outcome_label(df)` -> pd.DataFrame:
   - Add `home_win` column: 1 if `winning_team == home_team`, else 0
   - This is the prediction target

4. `_add_sp_features(df)` -> pd.DataFrame (FEAT-01, FEAT-06 partial):
   - For each season, load `fetch_sp_stats(season, min_gs=1)`
   - Build lookup: pitcher Name -> {FIP, xFIP, K%, SIERA, ERA} for that season
   - Join home SP stats: match `home_probable_pitcher` to sp_stats `Name` within same `season`
   - Join away SP stats: same for `away_probable_pitcher`
   - Compute differentials:
     - `sp_fip_diff = home_sp_fip - away_sp_fip`
     - `sp_xfip_diff = home_sp_xfip - away_sp_xfip`
     - `sp_k_pct_diff = home_sp_k_pct - away_sp_k_pct`
     - `sp_siera_diff = home_sp_siera - away_sp_siera` (FEAT-06)
   - If pitcher not found in sp_stats, features are NaN (acceptable per NaN policy). The join between MLB Stats API pitcher names (schedule `home_probable_pitcher`) and FanGraphs pitcher names (sp_stats `Name`) is a known mismatch source: diacritics may be stripped differently, name suffixes (Jr., II, III) may appear in one source and be absent from the other. Log all unmatched names at DEBUG level: `logger.debug(f"SP name unmatched: '{pitcher_name}' not found in sp_stats (season={season})")`. These mismatches are expected and produce NaN per the NaN policy, but logging them enables post-hoc auditing of match rates.

5. `_add_offense_features(df)` -> pd.DataFrame (FEAT-02):
   - For each season, load `fetch_team_batting(season)`
   - Build lookup: Team -> {wOBA, OPS, R, RA (if available)} for that season
   - Join home team stats on `(home_team, season)` matching team batting `Team` column
   - Join away team stats on `(away_team, season)`
   - Compute differentials:
     - `team_woba_diff = home_woba - away_woba`
     - `team_ops_diff = home_ops - away_ops`
   - Compute Pythagorean win% for home and away using `pythagorean_win_pct(R, RA)` (NOTE: R and RA columns -- check if team_batting provides these; if not available directly, they can be derived from schedule home_score/away_score aggregated per team per season)
   - `pyth_win_pct_diff = home_pyth - away_pyth`

6. `_add_rolling_features(df)` -> pd.DataFrame (FEAT-03):
   - For each season+team, load per-game batting log via `fetch_team_game_log(season, team)`
   - Concatenate all game logs
   - Sort by `["season", "team", "game_date"]`
   - Compute: `rolling_ops_10 = groupby(["team", "season"])["OPS"].transform(lambda x: x.shift(1).rolling(10, min_periods=10).mean())`
   - NOTE: If OPS is not a direct column in team_game_logs output, compute from OBP + SLG
   - NOTE: `min_periods=10` ensures games 1-9 are NaN (per CONTEXT.md -- no partial windows)
   - Join rolling stats to schedule on `(team, game_date)` for both home and away teams
   - `rolling_ops_diff = home_rolling_ops_10 - away_rolling_ops_10`

7. `_add_bullpen_features(df)` -> pd.DataFrame (FEAT-04, FEAT-06 partial):
   - For each season, load `fetch_sp_stats(season, min_gs=0)` -- qual=0 gets ALL pitchers
   - Filter pitchers to relievers: `GS < 3` (catches openers per RESEARCH.md) AND `IP >= 5` (excludes position players who pitched in blowouts -- their garbage-time ERA inflates team bullpen stats and is not representative of bullpen quality)
   - Aggregate by Team: mean ERA, mean FIP for qualifying relievers = team bullpen ERA, bullpen FIP
   - Join on `(home_team, season)` and `(away_team, season)`
   - Compute:
     - `bullpen_era_diff = home_bullpen_era - away_bullpen_era`
     - `bullpen_fip_diff = home_bullpen_fip - away_bullpen_fip` (FEAT-06)
   - NOTE: Need a separate `fetch_sp_stats(season, min_gs=0)` call for ALL pitchers (not just starters). The cache key will differ because min_gs=0 vs min_gs=1 -- this is handled by the existing sp_stats cache key pattern: `sp_stats_{season}_mings0`.

8. `_add_park_features(df)` -> pd.DataFrame (FEAT-05):
   - `is_home = 1` for all rows (this matrix is from home team perspective)
   - `park_factor = df["home_team"].map(get_park_factor)`

9. `_add_advanced_features(df)` -> pd.DataFrame (FEAT-06):
   - **xwOBA differential**: For each season, load `fetch_statcast_pitcher(season)`. Build pitcher -> xwOBA lookup. Join for home and away SP. `xwoba_diff = home_sp_xwoba - away_sp_xwoba`. NaN if no Statcast data.

   - **SP recent form (30-day rolling ERA via pitching_stats_range)** -- per CONTEXT.md locked decision:
     For each season in self.seasons:
       1. Collect all unique game dates for that season from the schedule DataFrame
       2. Call `fetch_sp_recent_form_bulk(game_dates, season)` from `src.features.sp_recent_form` to get 30-day pitcher stats for each date
       3. For each game, look up the home SP's ERA and away SP's ERA from the date-specific pitching_stats_range result:
          - Match `home_probable_pitcher` to the `Name` column in the pitching_stats_range result for that game's date
          - Match `away_probable_pitcher` similarly
       4. Compute: `sp_recent_era_diff = home_sp_recent_era - away_sp_recent_era`
       5. NaN if the pitcher does not appear in the 30-day window (e.g., pitcher made no starts in that window, or date range fetch failed). This is consistent with the NaN policy.

     Implementation detail: Build a dict of `{(game_date, pitcher_name): recent_era}` from the bulk fetch results, then map into the schedule DataFrame. This avoids N^2 DataFrame merges.

     IMPORTANT: The `fetch_sp_recent_form` function computes the window as `[game_date - 31 days, game_date - 1 day]` to ensure the game day itself is excluded (temporal safety). The column is named `sp_recent_era_diff` (not `sp_era_diff`) to clearly distinguish it from season-level ERA.

   - **Log5 win probability**: Must be derived from game-by-game schedule results with `shift(1)`, NOT from season-level batting data (which is a full-season aggregate, not a pre-game running total). Implementation:
     1. From the schedule DataFrame, derive each team's per-game result. For each game row, the home team won if `winning_team == home_team`. The away team won if `winning_team == away_team`.
     2. Build a long-form per-team game record: for each game, create two rows (one for each team) with columns `(team, season, game_date, won)`.
     3. Sort by `[team, season, game_date]`. Compute `cumulative_win_pct = wins.cumsum().shift(1) / games_played.shift(1)` per `(team, season)` group. Use `shift(1)` so a team's win% for a given game is computed from all PRIOR games in that season only.
     4. Default win% before any games played: 0.5 (neutral prior).
     5. Join cumulative win% back to the schedule DataFrame for both home and away teams.
     6. `log5_home_wp = log5_probability(home_cumulative_win_pct, away_cumulative_win_pct)`
     NOTE: Using a season-level W-L record (from batting stats) would include future games relative to any given date and constitute leakage.

10. `_add_kalshi_features(df)` -> pd.DataFrame (FEAT-08 partial):
    - Load `fetch_kalshi_markets()`
    - Type coercion MUST be explicit in the implementation code, not just in comments:
      ```python
      kalshi_df = fetch_kalshi_markets()
      kalshi_df['date'] = pd.to_datetime(kalshi_df['date'])
      df['game_date'] = pd.to_datetime(df['game_date'])
      kalshi_df = kalshi_df.rename(columns={'date': 'game_date'})
      ```
    - Left-join on `(game_date, home_team, away_team)` after type alignment
    - Keep `kalshi_yes_price` column (NaN for pre-2025 games -- expected behavior)

**Important implementation notes:**
- All rolling operations MUST use `shift(1)` within `groupby(["team", "season"])` -- this is NON-NEGOTIABLE per FEAT-07
- Rolling windows reset at season boundaries (groupby season ensures this)
- NaN policy: no imputation. NaN means "not available" and propagates cleanly.
- Log each step: `logger.info("Adding SP features...")` etc. for debugging.
- Use `import logging; logger = logging.getLogger(__name__)` at module level.
