Create `src/features/feature_builder.py` with a `FeatureBuilder` class. The class reads raw cached Parquet data from Phase 1 loaders and computes all differential features.

**Class signature:**
```python
class FeatureBuilder:
    def __init__(self, seasons: list[int], as_of_date: str | None = None):
        self.seasons = seasons
        self.as_of_date = as_of_date  # YYYY-MM-DD cutoff for walk-forward

    def build(self) -> pd.DataFrame:
        """Build complete feature matrix. Returns one row per game."""
```

**The `build()` method pipeline (each step is a private method):**

1. `_load_schedule()` -> pd.DataFrame:
   - For each season in self.seasons, call `fetch_schedule(season)`
   - Concatenate all seasons
   - If `self.as_of_date` is set, filter `game_date < self.as_of_date`
   - Ensure `game_date` is datetime type

2. `_filter_tbd_starters(df)` -> pd.DataFrame:
   - Drop rows where `home_probable_pitcher` is None/NaN OR `away_probable_pitcher` is None/NaN
   - Log count of excluded rows: `logger.info(f"Excluded {n} games with TBD starters")`

3. `_add_outcome_label(df)` -> pd.DataFrame:
   - Add `home_win` column: 1 if `winning_team == home_team`, else 0
   - This is the prediction target

4. `_add_sp_features(df)` -> pd.DataFrame (FEAT-01, FEAT-06 partial):
   - For each season, load `fetch_sp_stats(season, min_gs=1)`
   - Build lookup: pitcher Name -> {FIP, xFIP, K%, SIERA, ERA} for that season
   - Join home SP stats: match `home_probable_pitcher` to sp_stats `Name` within same `season`
   - Join away SP stats: same for `away_probable_pitcher`
   - Compute differentials:
     - `sp_fip_diff = home_sp_fip - away_sp_fip`
     - `sp_xfip_diff = home_sp_xfip - away_sp_xfip`
     - `sp_k_pct_diff = home_sp_k_pct - away_sp_k_pct`
     - `sp_siera_diff = home_sp_siera - away_sp_siera` (FEAT-06)
   - If pitcher not found in sp_stats, features are NaN (acceptable per NaN policy)

5. `_add_offense_features(df)` -> pd.DataFrame (FEAT-02):
   - For each season, load `fetch_team_batting(season)`
   - Build lookup: Team -> {wOBA, OPS, R, RA (if available)} for that season
   - Join home team stats on `(home_team, season)` matching team batting `Team` column
   - Join away team stats on `(away_team, season)`
   - Compute differentials:
     - `team_woba_diff = home_woba - away_woba`
     - `team_ops_diff = home_ops - away_ops`
   - Compute Pythagorean win% for home and away using `pythagorean_win_pct(R, RA)` (NOTE: R and RA columns -- check if team_batting provides these; if not available directly, they can be derived from schedule home_score/away_score aggregated per team per season)
   - `pyth_win_pct_diff = home_pyth - away_pyth`

6. `_add_rolling_features(df)` -> pd.DataFrame (FEAT-03):
   - For each season+team, load per-game batting log via `fetch_team_game_log(season, team)`
   - Concatenate all game logs
   - Sort by `["season", "team", "game_date"]`
   - Compute: `rolling_ops_10 = groupby(["team", "season"])["OPS"].transform(lambda x: x.shift(1).rolling(10, min_periods=10).mean())`
   - NOTE: If OPS is not a direct column in team_game_logs output, compute from OBP + SLG
   - NOTE: `min_periods=10` ensures games 1-9 are NaN (per CONTEXT.md -- no partial windows)
   - Join rolling stats to schedule on `(team, game_date)` for both home and away teams
   - `rolling_ops_diff = home_rolling_ops_10 - away_rolling_ops_10`

7. `_add_bullpen_features(df)` -> pd.DataFrame (FEAT-04, FEAT-06 partial):
   - For each season, load `fetch_sp_stats(season, min_gs=0)` -- qual=0 gets ALL pitchers
   - Actually, use the pitching_stats data to identify relievers: filter to GS < 3 (catches openers per RESEARCH.md)
   - Aggregate by Team: mean ERA, mean FIP for relievers = team bullpen ERA, bullpen FIP
   - Join on `(home_team, season)` and `(away_team, season)`
   - Compute:
     - `bullpen_era_diff = home_bullpen_era - away_bullpen_era`
     - `bullpen_fip_diff = home_bullpen_fip - away_bullpen_fip` (FEAT-06)
   - NOTE: Need a separate `fetch_sp_stats(season, min_gs=0)` call for ALL pitchers (not just starters). The cache key will differ because min_gs=0 vs min_gs=1 -- this is handled by the existing sp_stats cache key pattern: `sp_stats_{season}_mings0`.

8. `_add_park_features(df)` -> pd.DataFrame (FEAT-05):
   - `is_home = 1` for all rows (this matrix is from home team perspective)
   - `park_factor = df["home_team"].map(get_park_factor)`

9. `_add_advanced_features(df)` -> pd.DataFrame (FEAT-06):
   - **xwOBA differential**: For each season, load `fetch_statcast_pitcher(season)`. Build pitcher -> xwOBA lookup. Join for home and away SP. `xwoba_diff = home_sp_xwoba - away_sp_xwoba`. NaN if no Statcast data.
   - **SP recent form**: Load `fetch_sp_stats(season)` ERA column as a proxy. The CONTEXT.md decision was to use `pitching_stats_range(start_dt, end_dt)` for a 30-day rolling ERA window, but this requires date-range queries. Implementation approach: for each game, compute SP's ERA from the season-level stats (simpler) OR if the 30-day window is essential, call `pybaseball.pitching_stats_range()`. Use the simpler approach (season ERA) and name the column `sp_recent_era_diff` -- the column represents SP quality, and season ERA is a reasonable proxy for v1. Document that a true 30-day rolling window is a v2 enhancement.
   - **Log5 win probability**: Compute each team's season win% from schedule data (wins / games played that season so far, using shift(1) for temporal safety). Then `log5_home_wp = log5_probability(home_season_wp, away_season_wp)`.
   - `sp_era_diff = home_sp_era - away_sp_era` (SP quality via season ERA)

10. `_add_kalshi_features(df)` -> pd.DataFrame (FEAT-08 partial):
    - Load `fetch_kalshi_markets()`
    - Left-join on `(game_date == date, home_team, away_team)`
    - Keep `kalshi_yes_price` column (NaN for pre-2025 games -- expected behavior)
    - The date column in Kalshi data is named `date`; schedule uses `game_date`. Align before join.

**Important implementation notes:**
- All rolling operations MUST use `shift(1)` within `groupby(["team", "season"])` -- this is NON-NEGOTIABLE per FEAT-07
- Rolling windows reset at season boundaries (groupby season ensures this)
- NaN policy: no imputation. NaN means "not available" and propagates cleanly.
- Log each step: `logger.info("Adding SP features...")` etc. for debugging.
- Use `import logging; logger = logging.getLogger(__name__)` at module level.
