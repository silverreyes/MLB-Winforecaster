**Return type interface (define in-file, not exported):**
```typescript
interface EasternClockState {
  dateStr: string;   // e.g., "Monday, March 30"
  timeStr: string;   // e.g., "2:34 PM ET"
  nextUpdate: string; // e.g., "Next update: 1:00 PM ET" or "Next update: 10:00 AM ET tomorrow"
}
```

**Hook implementation:**

1. **State:** `useState<EasternClockState>` initialized by calling a pure `computeClock()` function.

2. **`computeClock()` pure function** (defined outside component, no side effects):
   - Create `new Date()` for current time.
   - **dateStr:** Format using `new Intl.DateTimeFormat('en-US', { weekday: 'long', month: 'long', day: 'numeric', timeZone: 'America/New_York' }).format(now)` -- produces "Monday, March 30".
   - **timeStr:** Format using `new Intl.DateTimeFormat('en-US', { hour: 'numeric', minute: '2-digit', timeZone: 'America/New_York' }).format(now)` + `' ET'` -- produces "2:34 PM ET".
   - **nextUpdate:** Call `computeNextUpdate(now)`.
   - Return `{ dateStr, timeStr, nextUpdate }`.

3. **`computeNextUpdate(now: Date): string` pure function:**
   - Convert `now` to Eastern Time hour/minute using `Intl.DateTimeFormat` with `hour: 'numeric', minute: '2-digit', hour12: false, timeZone: 'America/New_York'` and `formatToParts()` to extract numeric hour and minute.
   - Pipeline run times are `[10, 13, 17]` (10 AM, 1 PM, 5 PM ET).
   - Find the first run time where `runHour > currentHour || (runHour === currentHour && currentMinute < 0)` -- i.e., the earliest run hour strictly after the current ET hour. More precisely: compare `currentHour * 60 + currentMinute` against `runHour * 60` (runs are on the hour). The first run whose total minutes is strictly greater than current total minutes is the next update.
   - If a future run is found today: format as `"Next update: H:MM AM/PM ET"` where the time is formatted by creating a Date for today at that ET hour and formatting it (or simply use a lookup: 10 -> "10:00 AM", 13 -> "1:00 PM", 17 -> "5:00 PM"). A static lookup is simplest and avoids Date construction:
     ```typescript
     const RUN_LABELS: Record<number, string> = { 10: '10:00 AM', 13: '1:00 PM', 17: '5:00 PM' };
     ```
     Return `"Next update: " + RUN_LABELS[nextRunHour] + " ET"`.
   - If no future run today (current time is >= 17:00 ET): return `"Next update: 10:00 AM ET tomorrow"`.

4. **Drift-corrected interval in `useEffect`:**
   - On mount, calculate `msUntilNextSecond = 1000 - (Date.now() % 1000)`.
   - Set a `setTimeout` for `msUntilNextSecond` that:
     a. Calls `setState(computeClock())`.
     b. Starts a `setInterval(callback, 1000)` where `callback` does `setState(computeClock())`.
   - Store interval ID in a ref or variable for cleanup.
   - Cleanup: clear both timeout and interval on unmount.
   - This approach aligns the first tick to the wall-clock second boundary, then ticks every 1000ms. Since `computeClock()` always reads `new Date()`, any cumulative drift in setInterval only affects display latency by <1s, not accuracy (no accumulating time offset).
   - Dependency array: `[]` (mount once).

**Do NOT use `requestAnimationFrame` or any external library.** The setTimeout+setInterval pattern is sufficient since 1-second resolution does not require frame-level precision.
